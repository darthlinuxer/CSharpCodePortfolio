== Objetivo do nível

Este arquivo cobre modelagem aplicada: interfaces, classes abstratas, records, `virtual`, `override`, `sealed`, composição, associação, agregação, versionamento simples e escolhas de design que aparecem em código real.

== Herança, contratos e polimorfismo

.Quando usar uma classe abstrata?
[%collapsible]
====
Use classe abstrata quando existe uma base comum com comportamento compartilhado e partes que precisam ser especializadas.

Ela é adequada para uma família real de tipos.
Se o objetivo é apenas permitir substituição de implementação, uma interface pode ser melhor.
====

.Quando usar interface em vez de classe abstrata?
[%collapsible]
====
Use interface quando precisa definir um contrato sem impor hierarquia ou estado base.

Interfaces são boas para fronteiras entre camadas, plugins, adaptadores e testes.
Classes abstratas são melhores quando há estado ou algoritmo comum relevante.
====

.Métodos abstratos são virtuais?
[%collapsible]
====
Sim, no sentido de que devem ser implementados por tipos derivados.

A diferença é que um método `abstract` não tem implementação na classe base.
Um método `virtual` tem implementação base e pode ser sobrescrito.
====

.O que é `virtual`?
[%collapsible]
====
`virtual` permite que um membro seja sobrescrito por tipos derivados.

Use quando a classe base foi projetada para extensão naquele ponto.
Não marque tudo como virtual por padrão.
====

.O que é `override`?
[%collapsible]
====
`override` fornece a implementação de um membro virtual ou abstrato herdado.

O override deve preservar o contrato esperado do membro base.
Se muda a semântica de forma incompatível, há risco de violar LSP.
====

.O que é `new` em method hiding?
[%collapsible]
====
`new` esconde um membro herdado em vez de sobrescrevê-lo polimorficamente.

Isso pode confundir chamadas feitas por referência do tipo base.
Use raramente e prefira corrigir a abstração quando possível.
====

.Qual é a diferença entre overriding e hiding?
[%collapsible]
====
Overriding mantém despacho polimórfico.
Hiding escolhe o membro conforme o tipo da referência em tempo de compilação.

Se `Base x = new Derivada()`, um método sobrescrito chama a derivada; um método escondido pode chamar a base.
====

.O que é `sealed`?
[%collapsible]
====
`sealed` impede herança de uma classe ou novo override de um método.

Use quando a extensão não é suportada, quando invariantes não podem ser preservados por herdeiros ou para deixar o contrato explícito.
====

.Quando herança vira problema?
[%collapsible]
====
Herança vira problema quando tipos derivados não conseguem cumprir o contrato do tipo base.

Sinais: overrides que lançam `NotSupportedException`, propriedades ignoradas, métodos com semântica diferente e hierarquias criadas só para reaproveitar código.
====

.Como validar polimorfismo em testes?
[%collapsible]
====
Crie testes que exercem o contrato pelo tipo base ou pela interface.

Se cada implementação exige tratamento especial no chamador, o polimorfismo não está entregando seu objetivo.
====

== Composição e relações

.O que é associação?
[%collapsible]
====
Associação indica que dois objetos se relacionam ou colaboram.

É o termo mais amplo.
Agregação e composição são formas mais específicas de associação.
====

.O que é agregação?
[%collapsible]
====
Agregação representa relação todo-parte em que a parte pode existir independentemente.

Exemplo: um departamento tem funcionários, mas funcionários existem fora daquele departamento.
O ciclo de vida não é totalmente controlado pelo todo.
====

.O que é composição em sentido estrito?
[%collapsible]
====
Composição representa relação todo-parte forte, em que a parte pertence ao todo.

Exemplo: `Pedido` e seus `ItensDoPedido`.
Remover o pedido geralmente remove seus itens.
====

.Como escolher entre associação, agregação e composição?
[%collapsible]
====
Pergunte quem controla o ciclo de vida e as invariantes.

Se só há colaboração, associação.
Se há relação todo-parte com independência, agregação.
Se a parte pertence ao todo e depende dele, composição.
====

.Por que composição sobre herança é uma recomendação comum?
[%collapsible]
====
Porque composição permite variar comportamento sem congelar uma hierarquia.

Ela facilita testes, substituição e reuso.
Herança continua válida quando a relação "é um" é verdadeira e estável.
====

.O que é um objeto de valor?
[%collapsible]
====
Objeto de valor é definido por seus dados, não por identidade própria.

Exemplos: `Email`, `Money`, `Endereco`.
Em C#, records podem ser úteis para esse tipo de modelagem.
====

.O que é uma entidade?
[%collapsible]
====
Entidade tem identidade e ciclo de vida.

Duas entidades podem ter os mesmos dados e ainda serem diferentes por identidade.
Classes costumam ser uma escolha natural para entidades.
====

.Quando usar record, class ou struct?
[%collapsible]
====
Use record para valores imutáveis ou mensagens.
Use class para entidades, serviços e objetos com identidade.
Use struct para valores pequenos, imutáveis e sensíveis a alocação.

A escolha deve refletir semântica, não apenas sintaxe curta.
====

.O que são nested classes?
[%collapsible]
====
São tipos declarados dentro de outro tipo.

Use quando o tipo interno só faz sentido como detalhe do tipo externo.
Evite quando a estrutura dificulta leitura ou testes.
====

.Nested classes acessam membros privados do tipo externo?
[%collapsible]
====
Sim, nested classes podem acessar membros privados do tipo que as contém.

Elas não recebem automaticamente uma instância do tipo externo.
Para acessar membros de instância, precisam de uma referência a uma instância.
====

== Interfaces modernas e evolução

.O que são default interface methods?
[%collapsible]
====
São implementações padrão dentro de interfaces.

Elas ajudam a evoluir contratos sem quebrar todas as implementações.
Use com cuidado: muita lógica em interface pode esconder design ruim.
====

.Interfaces podem ter membros privados?
[%collapsible]
====
Sim, quando usados para apoiar implementações padrão dentro da própria interface.

Membros privados em interface não são acessíveis pela classe implementadora.
Eles servem para reduzir duplicação interna da interface.
====

.O que são static abstract interface members?
[%collapsible]
====
São membros estáticos exigidos por uma interface.

Eles permitem abstrações sobre operadores e membros estáticos, como em matemática genérica.
É um recurso moderno e deve ser usado quando há ganho real de tipo e performance.
====

.Como evoluir uma interface sem quebrar consumidores?
[%collapsible]
====
Prefira adicionar uma nova interface menor, criar extensão, usar implementação padrão quando apropriado ou versionar o contrato.

Adicionar membro abstrato a interface existente quebra implementações.
Contrato público precisa de cuidado de compatibilidade.
====

.Quando uma interface é abstração demais?
[%collapsible]
====
Quando existe uma única implementação, nenhum consumidor alternativo, nenhum boundary externo e nenhum ganho de teste.

Interface criada só por hábito aumenta arquivos e indireção.
Use abstração quando ela resolve uma variação real.
====

.O que é coesão em uma interface?
[%collapsible]
====
É o quanto seus membros pertencem ao mesmo contrato.

Interface coesa representa uma capacidade clara.
Interface com métodos desconexos força implementadores e consumidores a dependerem de membros irrelevantes.
====

.O que é acoplamento temporal?
[%collapsible]
====
É quando métodos precisam ser chamados em ordem específica para o objeto funcionar.

Exemplo: criar objeto, chamar `Configurar`, depois `Inicializar`, depois `Executar`.
Construtores, factories ou tipos de estado podem reduzir esse risco.
====

.Como construtores ajudam invariantes?
[%collapsible]
====
Construtores podem exigir dados obrigatórios e impedir objeto inválido.

Se um tipo não pode existir sem `Nome` e `Email`, esses valores devem ser obrigatórios na criação.
Isso simplifica o restante do código.
====

.Qual é a diferença entre validação e invariante?
[%collapsible]
====
Validação verifica entrada ou comando.
Invariante é regra que deve permanecer verdadeira durante toda a vida do objeto.

Domínio rico mantém invariantes perto do estado que elas protegem.
====

.Como reconhecer uma classe com responsabilidades demais?
[%collapsible]
====
Ela muda por muitos motivos, tem muitos grupos de métodos, conhece detalhes de várias camadas e é difícil de testar.

Nomes genéricos como `Manager`, `Helper` e `Service` podem ser sinal de baixa coesão, mas o diagnóstico vem das responsabilidades reais.
====

== Encapsulamento aplicado

.Como encapsular coleções em uma entidade?
[%collapsible]
====
Evite expor uma lista mutável quando qualquer código externo pode alterar o estado sem regra.

Uma alternativa comum é manter `List<T>` privada e expor `IReadOnlyCollection<T>`.
As mudanças entram por métodos com nomes de domínio, como `AdicionarItem`.
====

.Quando usar métodos estáticos nomeados para criação?
[%collapsible]
====
Use quando o construtor ficaria ambíguo ou quando há formas de criação com intenção diferente.

Exemplo: `Pedido.CriarRascunho()` e `Pedido.CriarConfirmado()`.
Isso ajuda leitura e concentra validação inicial.
====

.Como evitar acoplamento por DTO compartilhado?
[%collapsible]
====
Não use o mesmo tipo para API, domínio, persistência e mensageria quando cada fronteira muda por motivos diferentes.

Mapear tipos pode parecer repetição, mas reduz acoplamento entre contratos externos e modelo interno.
====

.Quando um método pertence à entidade e não ao serviço?
[%collapsible]
====
Quando a operação muda o estado da entidade e depende diretamente de suas invariantes.

Serviços devem coordenar dependências externas e casos de uso.
Regras que protegem o próprio objeto ficam mais claras dentro do modelo.
====
