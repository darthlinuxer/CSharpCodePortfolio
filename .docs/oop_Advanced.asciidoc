== Objetivo do nível

Este arquivo cobre decisões difíceis de modelagem: contratos comportamentais, LSP, versionamento de APIs, variância, imutabilidade, DDD, igualdade, pattern matching, limites arquiteturais, testabilidade e trade-offs entre abstração e simplicidade.

== Contratos e substituição

.Como detectar violação de LSP em uma hierarquia?
[%collapsible]
====
Procure subtipos que não conseguem cumprir promessas do tipo base.

Sinais: override lançando `NotSupportedException`, pré-condições mais fortes, pós-condições mais fracas, invariantes quebradas ou comportamento que obriga o chamador a testar o tipo concreto.
====

.Por que `NotSupportedException` em override costuma indicar problema de modelagem?
[%collapsible]
====
Porque o tipo base prometeu uma operação que o subtipo não consegue cumprir.

Às vezes a exceção é aceitável em APIs de propósito geral, mas em domínio próprio geralmente indica abstração errada.
Separe capacidades em contratos menores.
====

.Como preservar invariantes em tipos derivados?
[%collapsible]
====
Defina invariantes no tipo base e deixe pontos de extensão pequenos e controlados.

Evite permitir que derivados alterem estado essencial sem validação.
Se a classe base não consegue proteger suas regras, talvez não deva ser extensível.
====

.Quando selar uma classe pública?
[%collapsible]
====
Sele quando você não consegue garantir contrato para herdeiros ou não quer suportar extensão como API pública.

Classes públicas não seladas criam superfície de compatibilidade maior.
Permitir herança é assumir responsabilidade sobre comportamento de tipos derivados.
====

.Como desenhar uma classe base versionável?
[%collapsible]
====
Mantenha poucos membros protegidos, documente pontos de extensão e preserve ordem de chamadas.

Evite chamar métodos virtuais em construtores.
Mudanças em classe base podem quebrar derivados mesmo sem alterar assinatura pública.
====

.Qual é o risco de chamar método virtual no construtor?
[%collapsible]
====
O método derivado pode executar antes de o construtor derivado inicializar seu estado.

Isso cria objetos parcialmente construídos observáveis.
Prefira hooks explícitos após construção ou composição.
====

.Como testar contratos de uma hierarquia?
[%collapsible]
====
Crie uma suíte de contrato que execute os mesmos cenários para todas as implementações.

O teste deve usar o tipo base ou interface.
Se cada subtipo precisa de exceções no teste, o contrato está mal definido.
====

.Quando substituir herança por composição?
[%collapsible]
====
Quando a hierarquia muda por combinações de comportamento, políticas ou infraestrutura.

Composição permite combinar capacidades sem criar explosão de subclasses.
Herança é melhor para relações estáveis de substituição.
====

.Como evitar hierarquias anêmicas?
[%collapsible]
====
Coloque comportamento próximo dos dados que ele protege.

Se classes só têm propriedades e toda regra fica em serviços externos, invariantes ficam espalhadas.
O equilíbrio é manter regra de domínio no modelo e infraestrutura fora dele.
====

.Como lidar com requisitos que quebram a abstração atual?
[%collapsible]
====
Não force o novo caso na hierarquia existente.

Reavalie o contrato, extraia capacidades menores ou use composição.
O custo de mudar a abstração agora pode ser menor que acumular exceções e flags.
====

== Variance, igualdade e imutabilidade

.O que é covariance em interfaces genéricas?
[%collapsible]
====
Covariance permite usar um tipo mais específico onde se espera um tipo mais geral.

Em C#, `out` marca parâmetros usados como saída.
Exemplo: `IEnumerable<string>` pode ser tratado como `IEnumerable<object>`.
====

.O que é contravariance em delegates e interfaces?
[%collapsible]
====
Contravariance permite usar consumidor de tipo mais geral onde se espera consumidor de tipo mais específico.

Em C#, `in` marca parâmetros usados como entrada.
Exemplo: `Action<object>` pode ser usado como `Action<string>`.
====

.Por que variance não funciona com classes genéricas como `List<T>`?
[%collapsible]
====
Porque `List<T>` permite entrada e saída de `T`.

Se `List<string>` fosse `List<object>`, alguém poderia adicionar um `object` que não é string.
Interfaces somente leitura, como `IEnumerable<out T>`, podem ser covariantes com segurança.
====

.Como records implementam igualdade?
[%collapsible]
====
Records implementam igualdade baseada em valor por padrão.

Dois records do mesmo tipo runtime com os mesmos componentes são iguais.
Em hierarquias de records, o tipo runtime participa da igualdade.
====

.Quando igualdade por valor é perigosa?
[%collapsible]
====
Quando o objeto representa entidade com identidade própria.

Dois clientes com mesmo nome não são necessariamente o mesmo cliente.
Use igualdade por valor para value objects; use identidade para entidades.
====

.Como projetar value objects imutáveis?
[%collapsible]
====
Exija dados válidos na criação, não exponha setters mutáveis e implemente igualdade por valor.

Records podem ajudar, mas invariantes ainda precisam ser explícitas.
Um `Email` inválido não deve existir como objeto válido.
====

.Quando usar `required` e `init`?
[%collapsible]
====
Use para exigir inicialização sem abrir mutabilidade permanente.

Eles são úteis em DTOs e modelos de configuração.
Para domínio com invariantes fortes, construtores ou factories podem expressar melhor a criação válida.
====

.Como imutabilidade ajuda concorrência?
[%collapsible]
====
Objetos imutáveis podem ser compartilhados sem risco de alteração simultânea.

Isso reduz locks e efeitos colaterais.
O custo pode ser mais alocação ou necessidade de cópias estruturais.
====

.Quando mutabilidade é aceitável?
[%collapsible]
====
Quando há ciclo de vida claro, dono único ou ganho de performance medido.

Entidades de domínio frequentemente mudam estado por métodos controlados.
Mutabilidade pública sem regra costuma ser o problema, não a mutabilidade em si.
====

.Como modelar estado sem expor setters públicos?
[%collapsible]
====
Use métodos de intenção, como `Confirmar`, `Cancelar` e `AdicionarItem`.

Esses métodos validam transições e preservam invariantes.
Setters públicos deixam qualquer código montar estados inválidos.
====

== Modelagem avançada e limites

.Quando pattern matching substitui polimorfismo?
[%collapsible]
====
Pattern matching é bom quando o conjunto de casos é pequeno, fechado e local.

Polimorfismo é melhor quando o comportamento deve crescer por novas implementações.
Se todo `switch` precisa mudar ao adicionar tipo, talvez falte polimorfismo.
====

.Quando polimorfismo atrapalha mais do que ajuda?
[%collapsible]
====
Quando cria muitos tipos pequenos sem variação real ou esconde fluxo simples.

Se há dois casos estáveis e a lógica é local, um `switch` claro pode ser melhor.
Abstração deve reduzir complexidade, não apenas trocar `if` por arquivos.
====

.Como aplicar DDD sem supermodelar?
[%collapsible]
====
Modele invariantes e linguagem do domínio onde há regra real.

Não crie agregado, repository, factory e domain service para CRUD simples sem comportamento.
Comece pelo domínio que muda e dói.
====

.O que é agregado em DDD?
[%collapsible]
====
Agregado é um conjunto de objetos tratado como unidade de consistência.

Ele tem uma raiz que controla acesso e preserva invariantes.
Nem todo relacionamento entre entidades precisa estar no mesmo agregado.
====

.Como escolher limites entre domínio e infraestrutura?
[%collapsible]
====
Domínio deve conhecer regras e linguagem de negócio.
Infraestrutura deve lidar com banco, HTTP, arquivos, fila e frameworks.

Quando domínio depende diretamente de detalhes de infraestrutura, testes e evolução ficam mais caros.
====

.Como evitar uma camada de serviços anêmica?
[%collapsible]
====
Não coloque todas as regras em serviços proceduralmente enquanto entidades ficam passivas.

Use serviços para orquestração e dependências externas.
Coloque invariantes e transições no modelo quando elas pertencem ao estado.
====

.Como avaliar se uma abstração deve existir?
[%collapsible]
====
Pergunte qual variação ela protege, qual consumidor precisa dela e qual teste ou boundary melhora.

Se não há resposta concreta, a abstração é especulativa.
Código simples sem interface pode ser melhor até existir pressão real.
====

.Como modelar erros de domínio?
[%collapsible]
====
Erros esperados do domínio podem ser resultados explícitos.
Falhas inesperadas podem ser exceções.

O importante é não misturar validação comum com exceções genéricas sem contexto.
====

.Como projetar APIs testáveis sem expor detalhes internos?
[%collapsible]
====
Teste comportamento público e contratos observáveis.

Use dependências injetadas nas fronteiras externas.
Evite tornar tudo público apenas para teste; quando necessário, use `internal` com critério.
====

.Como revisar acoplamento em OOP avançado?
[%collapsible]
====
Observe direção das dependências, tipos concretos atravessando camadas, objetos conhecendo detalhes demais e mudanças que se propagam.

Baixo acoplamento não significa ausência de dependência.
Significa dependências explícitas, estáveis e justificadas.
====

== Consistência em sistemas reais

.Como proteger invariantes em cenários concorrentes?
[%collapsible]
====
Invariantes de objeto não bastam quando duas operações concorrentes leem o mesmo estado e gravam decisões conflitantes.

Use transações, controle de concorrência, versionamento otimista ou comandos idempotentes conforme o caso.
O modelo precisa ser consistente com a fronteira de persistência.
====

.Como decidir o limite transacional de um agregado?
[%collapsible]
====
Coloque no mesmo agregado o que precisa mudar de forma atomicamente consistente.

Não use relacionamento ou conveniência de navegação como critério principal.
Agregados grandes reduzem inconsistência local, mas aumentam contenção e custo de carga.
====

.Como manter compatibilidade binária em uma biblioteca OOP?
[%collapsible]
====
Evite remover membros públicos, mudar assinaturas, trocar tipos de retorno ou alterar comportamento documentado sem versionamento.

Adicionar método abstrato em classe base pública também quebra consumidores.
Design avançado de APIs considera consumidores que já compilaram contra versões anteriores.
====

.Como distinguir regra de domínio de regra de aplicação?
[%collapsible]
====
Regra de domínio continua verdadeira independentemente de interface, banco ou caso de uso específico.
Regra de aplicação coordena fluxo, autorização, transação, integração e decisão operacional.

Misturar tudo no domínio cria acoplamento externo.
Tirar tudo do domínio gera modelo anêmico.
====
