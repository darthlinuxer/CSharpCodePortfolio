= SOLID em C# - Perguntas por nível
:toc:
:toc-title: Sumário

== Objetivo

Este arquivo cobre SOLID como ferramenta de análise de design, não como lista decorativa de siglas. As perguntas avançam de definição para revisão de código, refatoração e trade-offs com simplicidade.

== Fundamentos

.Qual é o objetivo de SOLID?
[%collapsible]
====
SOLID ajuda a reduzir acoplamento, aumentar coesão e tornar mudanças menos arriscadas.

Os princípios não são regras mecânicas.
Eles servem para orientar decisões de design quando o código precisa evoluir.
====

.O que é SRP?
[%collapsible]
====
SRP significa Single Responsibility Principle.

Uma unidade deve ter um motivo claro de mudança.
Na prática, pergunte quais atores ou regras podem exigir alteração naquela classe.
====

.O que é OCP?
[%collapsible]
====
OCP significa Open/Closed Principle.

O código deve ser aberto para extensão e fechado para modificação.
Isso não significa criar abstrações para tudo, mas evitar editar código estável a cada novo caso previsto.
====

.O que é LSP?
[%collapsible]
====
LSP significa Liskov Substitution Principle.

Um subtipo deve poder substituir seu tipo base sem quebrar expectativas do consumidor.
O foco é comportamento, não apenas herança que compila.
====

.O que é ISP?
[%collapsible]
====
ISP significa Interface Segregation Principle.

Consumidores não devem depender de membros que não usam.
Interfaces pequenas e coesas reduzem implementações falsas e mocks ruidosos.
====

.O que é DIP?
[%collapsible]
====
DIP significa Dependency Inversion Principle.

Políticas de alto nível não devem depender diretamente de detalhes de baixo nível.
Ambas devem depender de abstrações orientadas ao caso de uso.
====

.Qual é a relação entre SOLID e YAGNI?
[%collapsible]
====
YAGNI evita construir variação que ainda não existe.
SOLID ajuda a organizar variação real.

O equilíbrio é não criar abstrações especulativas, mas também não ignorar acoplamentos que já causam dor.
====

== SRP

.Como identificar violação de SRP?
[%collapsible]
====
Procure classes que mudam por motivos diferentes.

Exemplo: uma classe que calcula preço, salva no banco, envia e-mail e formata HTML tem responsabilidades de domínio, persistência, integração e apresentação misturadas.
====

.Como refatorar uma classe que viola SRP?
[%collapsible]
====
Separe responsabilidades por motivo de mudança.

Regras de negócio ficam no domínio ou serviço de aplicação.
Persistência vai para repositório ou gateway.
Notificação vai para componente próprio.
Formatação vai para apresentação.
====

.Quando dividir por SRP pode piorar o design?
[%collapsible]
====
Quando a divisão cria tipos artificiais sem responsabilidade real.

Se cada método vira uma classe, o fluxo fica espalhado e difícil de ler.
Divida quando há mudança independente, não por estética.
====

== OCP

.Como identificar violação de OCP?
[%collapsible]
====
Procure `if`, `switch` ou listas de casos que precisam ser editados sempre que um novo tipo de regra aparece.

Isso é problema quando os casos mudam com frequência e o código existente já é estável.
Se há poucos casos fixos, um `switch` pode ser mais simples.
====

.Como aplicar OCP sem criar abstração especulativa?
[%collapsible]
====
Espere uma variação real ou muito provável.

Quando ela aparecer, extraia o ponto de extensão menor possível.
Não crie plugin, factory e registry para dois casos locais e estáveis.
====

.Quando strategy ajuda OCP?
[%collapsible]
====
Strategy ajuda quando há várias políticas intercambiáveis com o mesmo contrato.

Exemplo: cálculo de frete por transportadora.
O chamador usa `ICalculoFrete`, e novas estratégias entram sem alterar o fluxo principal.
====

.Quando pattern matching é melhor que OCP via polimorfismo?
[%collapsible]
====
Quando o conjunto de casos é fechado, pequeno e local.

Pattern matching deixa a regra explícita em um ponto.
Polimorfismo é melhor quando novos comportamentos surgem em módulos independentes.
====

== LSP

.Como revisar LSP em código C#?
[%collapsible]
====
Use o tipo base no teste e substitua por cada derivado.

Se um derivado exige tratamento especial, lança exceção para método herdado ou muda semântica esperada, há risco de violação.
====

.Como pré-condições e pós-condições se relacionam com LSP?
[%collapsible]
====
Um subtipo não deve exigir pré-condições mais fortes que o tipo base.
Também não deve entregar pós-condições mais fracas.

Se o consumidor precisa saber o subtipo para chamar com segurança, a substituição falhou.
====

.LSP vale para interfaces?
[%collapsible]
====
Sim.

Uma classe que implementa interface deve cumprir o contrato comportamental esperado, não apenas assinar métodos.
Interface com métodos genéricos demais costuma gerar implementações que violam LSP e ISP.
====

.Como corrigir violação de LSP?
[%collapsible]
====
Reveja a hierarquia.

Opções comuns: separar interfaces por capacidade, substituir herança por composição, remover método do tipo base ou criar tipos de domínio mais precisos.
====

== ISP

.Como identificar violação de ISP?
[%collapsible]
====
Procure classes que implementam métodos vazios, lançam `NotSupportedException` ou têm mocks cheios de membros irrelevantes.

Isso indica que consumidores dependem de contratos maiores do que precisam.
====

.Qual é o risco de interfaces pequenas demais?
[%collapsible]
====
Fragmentação excessiva.

Se cada método vira uma interface, a composição fica ruidosa.
O ideal é uma interface coesa por capacidade percebida pelo consumidor.
====

.Quem deve definir a interface: consumidor ou implementação?
[%collapsible]
====
Normalmente o consumidor.

A abstração deve representar o que o caso de uso precisa, não espelhar todos os métodos da implementação concreta.
Isso evita wrappers sem valor.
====

.Quando separar leitura e escrita em interfaces?
[%collapsible]
====
Quando consumidores diferentes precisam de capacidades diferentes.

Exemplo: um relatório pode precisar apenas de leitura; um comando precisa de escrita.
Separar evita dar poder desnecessário ao consumidor.
====

== DIP, DI e arquitetura

.Qual é a diferença entre DIP e dependency injection?
[%collapsible]
====
DIP é princípio de design.
Dependency injection é uma técnica para fornecer dependências.

Você pode usar DI sem respeitar DIP se injetar detalhes concretos por toda parte.
====

.O que é composition root?
[%collapsible]
====
É o ponto onde a aplicação monta suas dependências.

Em ASP.NET Core, normalmente fica na inicialização.
Domínio e casos de uso não devem conhecer o container diretamente.
====

.Por que service locator costuma ser problemático?
[%collapsible]
====
Porque esconde dependências.

Em vez de o construtor declarar o que precisa, o código busca dependências em um contêiner global.
Isso dificulta testes, leitura e análise de impacto.
====

.Como DIP se aplica a infraestrutura?
[%collapsible]
====
Regras de negócio devem depender de contratos, não de banco, HTTP, fila ou sistema de arquivos diretamente.

Adaptadores de infraestrutura implementam contratos definidos pelo caso de uso.
Assim a política central não muda por detalhe externo.
====

.Quando uma interface para DIP é desnecessária?
[%collapsible]
====
Quando não há fronteira externa, variação real, teste difícil ou política de alto nível protegida.

Criar `IFoo` para toda classe concreta pode ser só ruído.
DIP não exige interface para cada tipo.
====

== Revisão e cenários

.Como revisar uma classe `OrderService` grande usando SOLID?
[%collapsible]
====
Identifique responsabilidades: cálculo, validação, persistência, pagamento, notificação e logs.

Depois separe por motivo de mudança e por fronteira externa.
Nem tudo precisa virar interface; comece pelos acoplamentos que dificultam teste e evolução.
====

.Como aplicar SOLID em código com EF Core?
[%collapsible]
====
Não esconda EF Core automaticamente atrás de repository genérico sem motivo.

Use abstrações quando o domínio ou caso de uso precisa ignorar persistência.
Mantenha queries específicas onde elas são mais claras e testáveis.
====

.Como aplicar SOLID em Minimal APIs?
[%collapsible]
====
Endpoints devem coordenar entrada e saída HTTP.

Regras de negócio ficam em serviços ou domínio.
Validação, autenticação, persistência e resposta devem ser claras sem transformar cada endpoint em uma arquitetura pesada.
====

.Quando quebrar um princípio SOLID é aceitável?
[%collapsible]
====
Quando a alternativa adiciona complexidade maior que o problema resolvido.

Código pequeno, estável e local pode ser direto.
A decisão deve ser consciente: documente o trade-off quando houver risco futuro claro.
====

.Como saber se uma refatoração SOLID deu certo?
[%collapsible]
====
Mudanças novas ficam mais localizadas, testes ficam mais simples e o fluxo continua legível.

Se a refatoração aumenta arquivos, indireção e setup sem reduzir risco, ela provavelmente passou do ponto.
====
