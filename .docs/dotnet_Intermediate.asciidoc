== Objetivo do nível

Este arquivo cobre recursos que exigem experiência prática: delegates, eventos, generics, LINQ com avaliação adiada, assincronismo com cancelamento, DI, testes, reflection e APIs modernas de C#.

== Linguagem e tipos

.Quando usar delegates em C#?
[%collapsible]
====
Use delegates para representar métodos como valores.

Eles aparecem em callbacks, eventos, LINQ, `Func<>`, `Action<>`, estratégias simples e pontos de extensão.
Em código moderno, normalmente usamos `Func<T>`, `Action<T>` ou eventos em vez de declarar delegates próprios.
====

.Qual é a diferença entre delegate e event?
[%collapsible]
====
Delegate é o tipo que representa uma chamada.
Event é uma camada de encapsulamento para publicação e assinatura.

Com `event`, consumidores externos podem assinar e desassinar, mas não podem disparar o evento diretamente.
====

.O que é um multicast delegate?
[%collapsible]
====
É um delegate com uma lista de invocação contendo mais de um método.

Ao invocar, os handlers são chamados em sequência.
Esse comportamento é a base de muitos eventos em .NET.
====

.Como tratar exceções em eventos multicast?
[%collapsible]
====
Se um handler lança exceção, a invocação direta pode interromper os próximos handlers.

Quando cada assinante precisa ser isolado, percorra `GetInvocationList()` e trate falhas individualmente.
Isso é importante em infraestrutura, plugins e notificações internas.
====

.O que são generics?
[%collapsible]
====
Generics permitem escrever tipos e métodos parametrizados por tipo.

Eles preservam segurança de tipos, reduzem casts, evitam boxing em muitos cenários e deixam APIs reutilizáveis.
Exemplo: `List<string>` e `Dictionary<int, Cliente>`.
====

.Para que servem constraints em generics?
[%collapsible]
====
Constraints limitam os tipos aceitos por um parâmetro genérico.

Exemplos: `where T : class`, `where T : struct`, `where T : new()`, `where T : IDisposable`.
Elas permitem usar membros e garantias específicas dentro do código genérico.
====

.Qual é a diferença entre covariance e contravariance?
[%collapsible]
====
Covariance preserva compatibilidade do tipo mais específico para o mais geral, como `IEnumerable<string>` para `IEnumerable<object>`.

Contravariance inverte a direção para entradas, como `Action<object>` podendo ser usado onde se espera `Action<string>`.
Em C#, interfaces e delegates podem declarar isso com `out` e `in`.
====

.O que são extension methods?
[%collapsible]
====
São métodos estáticos que parecem métodos de instância.

Eles ajudam a adicionar operações a tipos existentes sem herança.
LINQ usa extension methods extensivamente.
====

.Quando usar records?
[%collapsible]
====
Use records para modelos orientados a valor, especialmente DTOs, mensagens, resultados e objetos imutáveis simples.

Records fornecem igualdade por valor, `with` expressions e uma sintaxe concisa.
Para entidades com identidade e ciclo de vida, classes costumam ser mais adequadas.
====

.O que é pattern matching?
[%collapsible]
====
Pattern matching permite testar forma, tipo e propriedades de valores.

Ele aparece em `is`, `switch`, padrões relacionais, padrões de propriedade e padrões posicionais.
O benefício é expressar decisões complexas de forma declarativa e legível.
====

== LINQ, coleções e dados

.Como deferred execution pode causar bugs?
[%collapsible]
====
Uma consulta LINQ pode capturar a fonte e executar só depois.

Se a fonte muda antes da enumeração, o resultado muda.
Se a consulta acessa banco ou rede, múltiplas enumerações podem repetir trabalho.
Materialize com `ToList()` quando precisar congelar o resultado.
====

.Quando evitar múltipla enumeração de `IEnumerable<T>`?
[%collapsible]
====
Evite quando a sequência é cara, depende de I/O, tem efeitos colaterais ou só pode ser lida uma vez.

Chamadas como `Count()`, `Any()` e `foreach` podem reexecutar a consulta.
Se necessário, materialize uma vez.
====

.Qual é a diferença entre `Select` e `SelectMany`?
[%collapsible]
====
`Select` projeta cada item para um resultado.
`SelectMany` projeta cada item para uma sequência e achata o resultado.

Use `SelectMany` para transformar uma lista de listas em uma sequência única.
====

.Como escolher entre `Any`, `Count` e `FirstOrDefault`?
[%collapsible]
====
Use `Any` para saber se existe ao menos um item.
Use `Count` quando precisa da quantidade total.
Use `FirstOrDefault` quando precisa do primeiro valor ou de um padrão.

Para existência, `Any` costuma ser mais eficiente porque pode parar no primeiro item.
====

.O que é `IQueryable<T>`?
[%collapsible]
====
`IQueryable<T>` representa uma consulta que pode ser traduzida por um provedor, como Entity Framework Core.

Expressões LINQ podem virar SQL ou outra linguagem de consulta.
Isso é poderoso, mas nem todo código C# pode ser traduzido.
====

== Assincronismo e concorrência

.Como funciona `CancellationToken`?
[%collapsible]
====
`CancellationToken` comunica pedido de cancelamento cooperativo.

Ele não mata a operação à força.
O código precisa observar o token, repassá-lo para APIs assíncronas e parar de forma consistente.
====

.Quando usar `Task.WhenAll`?
[%collapsible]
====
Use quando várias operações assíncronas independentes podem iniciar juntas.

Ele aguarda todas terminarem e agrega falhas.
É adequado para fan-out de I/O, desde que o volume não sobrecarregue o recurso externo.
====

.Quando limitar concorrência?
[%collapsible]
====
Limite concorrência quando há risco de saturar banco, HTTP, CPU, memória, fila ou serviço externo.

Ferramentas comuns incluem `SemaphoreSlim`, `Parallel.ForEachAsync`, channels ou filas.
O objetivo é controlar pressão, não apenas iniciar tudo ao mesmo tempo.
====

.Qual é o risco de usar `.Result` ou `.Wait()` em código assíncrono?
[%collapsible]
====
Essas chamadas bloqueiam a thread e podem causar deadlocks em ambientes com contexto de sincronização.

Também prejudicam escalabilidade em servidores.
Prefira propagar `async` e usar `await`.
====

.O que é `ConfigureAwait(false)`?
[%collapsible]
====
Ele indica que a continuação não precisa voltar ao contexto capturado.

É comum em bibliotecas para evitar dependência de contexto de UI ou requisição.
Em ASP.NET Core moderno, o impacto é menor, mas ainda pode ser relevante em bibliotecas reutilizáveis.
====

== APIs, DI e testes

.O que é dependency injection?
[%collapsible]
====
Dependency injection é fornecer dependências de fora do tipo, em vez de criá-las internamente.

Isso reduz acoplamento, facilita testes e deixa o ciclo de vida das dependências sob controle do container.
====

.Qual é a diferença entre transient, scoped e singleton?
[%collapsible]
====
Transient cria uma nova instância a cada resolução.
Scoped reutiliza a instância dentro de um escopo, como uma requisição web.
Singleton cria uma instância para toda a aplicação.

O erro clássico é colocar dependência scoped dentro de singleton.
====

.Como testar código que usa `HttpClient`?
[%collapsible]
====
Evite mockar `HttpClient` diretamente.

Use um `HttpMessageHandler` customizado, um servidor de teste ou uma abstração pequena quando o domínio pedir.
Assim o teste controla resposta, status code, headers e falhas.
====

.O que é reflection?
[%collapsible]
====
Reflection permite inspecionar metadados e acessar tipos, membros e atributos em tempo de execução.

É útil para descoberta dinâmica, serialização, frameworks e ferramentas.
O custo é maior complexidade, menor segurança em tempo de compilação e impacto em trimming/AOT.
====

.Quando usar atributos personalizados?
[%collapsible]
====
Use atributos para metadados declarativos.

Eles são bons para marcação, validação, configuração leve e descoberta por reflection.
Não devem virar um segundo sistema de programação escondido.
====

.Qual é a diferença entre unit test e integration test?
[%collapsible]
====
Unit test valida uma unidade pequena com dependências controladas.
Integration test valida a colaboração entre componentes reais, como banco, HTTP, fila ou container DI.

Ambos são úteis; o ponto é saber qual risco cada teste cobre.
====

.O que é `InternalsVisibleTo`?
[%collapsible]
====
É um atributo que permite a outro assembly acessar membros `internal`.

É comum para expor detalhes internos a um projeto de testes sem tornar APIs públicas.
Use com moderação para não testar implementação demais.
====

.Como versionar uma API pública em biblioteca?
[%collapsible]
====
Evite remover membros ou mudar comportamento esperado sem versão maior.

Prefira adicionar overloads, manter compatibilidade binária quando possível, documentar obsolescência com `[Obsolete]` e cobrir contratos públicos com testes.
====

.Quando criar uma abstração própria?
[%collapsible]
====
Crie abstração quando houver variação real, teste difícil, fronteira externa ou contrato de domínio claro.

Não crie interface para uma única implementação sem motivo.
Abstração sem pressão real aumenta manutenção.
====

.Como lidar com opções de configuração em .NET?
[%collapsible]
====
Use o padrão Options quando há configuração tipada, validação e injeção.

Para valores simples e locais, uma constante ou objeto de configuração direto pode ser suficiente.
O importante é validar configuração em fronteiras de inicialização.
====

.O que observar ao usar EF Core em serviços?
[%collapsible]
====
`DbContext` representa unidade de trabalho e não é thread-safe.

Use ciclo de vida scoped em aplicações web, não compartilhe contexto entre threads e evite consultas que carreguem mais dados que o necessário.
====
