== Objetivo do nível

Este arquivo cobre a base de programação orientada a objetos em C#: classes, objetos, estado, comportamento, encapsulamento, construtores, modificadores de acesso, herança simples, composição e polimorfismo introdutório.

== Conceitos centrais

.O que é programação orientada a objetos?
[%collapsible]
====
Programação orientada a objetos organiza código em tipos que combinam estado e comportamento.

Em C#, isso aparece principalmente por meio de classes, objetos, interfaces, structs, records, propriedades e métodos.
O objetivo é modelar conceitos do domínio e controlar dependências entre partes do sistema.
====

.O que é uma classe?
[%collapsible]
====
Classe é um molde que define dados e comportamentos.

Ela descreve quais propriedades, campos, métodos e construtores seus objetos terão.
Exemplo: `Cliente` pode definir `Nome`, `Email` e `Ativar()`.
====

.O que é um objeto?
[%collapsible]
====
Objeto é uma instância concreta de uma classe.

Se `Cliente` é o tipo, `new Cliente(...)` cria um objeto específico.
Cada objeto pode ter seu próprio estado.
====

.Qual é a diferença entre estado e comportamento?
[%collapsible]
====
Estado é o conjunto de dados do objeto.
Comportamento são as operações que o objeto sabe executar.

Em uma conta bancária, saldo é estado; depositar e sacar são comportamentos.
====

.O que é encapsulamento?
[%collapsible]
====
Encapsulamento é proteger o estado interno e expor operações controladas.

Em vez de permitir alteração direta de qualquer campo, o tipo oferece métodos e propriedades que preservam regras.
Isso reduz inconsistência e facilita manutenção.
====

.O que é abstração?
[%collapsible]
====
Abstração é representar só o que importa para o problema.

Um tipo `Pedido` não precisa expor detalhes de banco de dados para aplicar regra de negócio.
Boa abstração mostra intenção e esconde detalhe.
====

.O que é herança?
[%collapsible]
====
Herança permite criar um tipo derivado a partir de um tipo base.

Ela expressa uma relação "é um".
Exemplo: `ContaCorrente` pode herdar de `Conta`, se realmente puder ser usada como `Conta`.
====

.O que é composição?
[%collapsible]
====
Composição é montar um objeto usando outros objetos.

Ela expressa relação "tem um" ou "usa um".
Exemplo: `Pedido` tem `ItensDoPedido`.
====

.O que é polimorfismo?
[%collapsible]
====
Polimorfismo permite tratar diferentes tipos por um contrato comum.

Exemplo: vários meios de pagamento podem implementar uma operação `Pagar`.
O chamador usa o contrato sem conhecer todos os detalhes de cada implementação.
====

.Quais são os quatro pilares de OOP?
[%collapsible]
====
Os quatro pilares geralmente citados são abstração, encapsulamento, herança e polimorfismo.

Eles são úteis como vocabulário inicial.
Na prática, bom design também depende de coesão, baixo acoplamento, composição e regras de domínio claras.
====

== C# básico aplicado a OOP

.O que é uma propriedade?
[%collapsible]
====
Propriedade é uma forma controlada de expor dados de um tipo.

Ela pode ter `get`, `set`, `init` ou lógica de validação.
Propriedades compõem a API do objeto.
====

.O que é um campo?
[%collapsible]
====
Campo é uma variável armazenada dentro do tipo.

Campos costumam ser privados e usados como detalhe interno.
Expor campo público geralmente quebra encapsulamento.
====

.O que é um método?
[%collapsible]
====
Método é uma operação que um tipo executa.

Métodos devem expressar comportamento, como `CalcularTotal`, `Ativar` ou `Enviar`.
Um método bom deixa claro o que muda e por quê.
====

.O que é um construtor?
[%collapsible]
====
Construtor inicializa um objeto.

Ele deve receber os dados obrigatórios para que o objeto nasça em estado válido.
Evite criar objetos que precisam de várias chamadas extras antes de poderem ser usados.
====

.O que é um construtor padrão?
[%collapsible]
====
É um construtor sem parâmetros.

Ele é útil para alguns frameworks e cenários simples.
Quando o objeto tem invariantes obrigatórios, um construtor com parâmetros pode ser mais adequado.
====

.O que significa `public`?
[%collapsible]
====
`public` permite acesso a partir de qualquer código que referencie o tipo.

Use para APIs que realmente fazem parte do contrato externo.
Tudo que é público tende a ser mais caro de mudar.
====

.O que significa `private`?
[%collapsible]
====
`private` restringe acesso ao próprio tipo.

É o modificador mais seguro para detalhes internos.
Comece privado e aumente visibilidade apenas quando houver necessidade real.
====

.O que significa `protected`?
[%collapsible]
====
`protected` permite acesso pelo tipo e por tipos derivados.

Use com cuidado: membros protegidos também viram contrato para herdeiros.
Se não há uma hierarquia clara, prefira `private`.
====

.O que significa `internal`?
[%collapsible]
====
`internal` permite acesso dentro do mesmo assembly.

É útil para APIs internas do projeto que não devem ser públicas para consumidores externos.
Testes podem acessar membros internos com `InternalsVisibleTo`.
====

.Qual é a diferença entre membro estático e membro de instância?
[%collapsible]
====
Membro de instância pertence a um objeto específico.
Membro estático pertence ao tipo.

Use estático para operações sem estado de instância ou valores compartilhados.
Evite estado estático mutável quando isso prejudica testes e concorrência.
====

== Relações entre objetos

.O que é relação "é um"?
[%collapsible]
====
É uma relação típica de herança.

Se `Gato` herda de `Animal`, deve ser seguro usar `Gato` onde se espera `Animal`.
Se isso não for verdade, a herança está modelando mal o domínio.
====

.O que é relação "tem um"?
[%collapsible]
====
É uma relação típica de composição ou agregação.

Um `Pedido` tem itens.
Um `Cliente` pode ter endereços.
Essa relação costuma ser mais flexível que herança.
====

.Quando preferir composição a herança?
[%collapsible]
====
Prefira composição quando você quer reutilizar comportamento sem criar uma família rígida de tipos.

Composição reduz acoplamento e evita heranças artificiais.
Herança deve representar substituição real, não apenas reaproveitamento de código.
====

.O que é coesão?
[%collapsible]
====
Coesão mede o quanto as responsabilidades de um tipo pertencem ao mesmo assunto.

Uma classe coesa tem motivo claro para existir.
Classe que valida, salva, renderiza e envia e-mail provavelmente tem responsabilidades demais.
====

.O que é acoplamento?
[%collapsible]
====
Acoplamento mede o quanto um código depende de outro.

Alto acoplamento torna mudanças caras.
Baixo acoplamento facilita testes, substituição e evolução.
====

.O que é uma interface em nível básico?
[%collapsible]
====
Interface define um contrato de comportamento.

Ela diz o que um tipo deve oferecer, sem obrigar a uma implementação específica.
Exemplo: `IClock` pode expor `Now`.
====

.Qual é a diferença entre classe e interface?
[%collapsible]
====
Classe pode ter estado, construtores e implementação completa.
Interface define contrato de membros que tipos podem implementar.

Use classe para modelar implementação ou entidade.
Use interface para contrato entre consumidor e implementação.
====

.O que é sobrescrita de método?
[%collapsible]
====
Sobrescrita ocorre quando um tipo derivado altera o comportamento de um método virtual do tipo base.

Em C#, isso usa `virtual` no método base e `override` no derivado.
Esse assunto começa no básico, mas seus riscos aparecem nos níveis intermediário e avançado.
====

.O que é sobrecarga de método?
[%collapsible]
====
Sobrecarga é ter métodos com o mesmo nome e assinaturas diferentes.

Exemplo: `Calcular(int valor)` e `Calcular(decimal valor)`.
Ela deve melhorar legibilidade, não esconder comportamentos incompatíveis.
====

.O que é responsabilidade de uma classe?
[%collapsible]
====
Responsabilidade é aquilo que a classe conhece e faz.

Uma boa classe tem responsabilidade clara e pequena o suficiente para ser entendida.
Quando uma classe muda por muitos motivos diferentes, ela provavelmente precisa ser dividida.
====
